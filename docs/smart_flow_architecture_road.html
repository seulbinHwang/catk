Detailed Polygon Semantic Encoder

Polygon Semantic Encoder를 더 디테일하게 풀어 쓴 구조

이번 개정안의 핵심은 여기입니다. 아래 구조는 polygon을 단순히 선분 조각으로 바꾸지 않고, 원래의 경계 모양, 객체 종류, 주변 도로와의 연결 관계를 작은 파라미터로 함께 담도록 만든 설계입니다.

새로 추가되는 branch · 대상: crosswalk / speed bump / driveway entrance

입력과 출력

이 branch는 road token bank를 전혀 거치지 않습니다. 처음부터 polygon 전용 표현으로 들어가고, 마지막에만 road token과 합칩니다.

1) polygon canonicalizer

New / Geometry Setup

먼저 원래 polygon 경계를 그대로 읽고, 객체 중심을 기준으로 local frame을 만듭니다. 여기서 가장 중요한 선택은 local x축을 긴 축이 아니라 짧은 축 방향으로 두는 것입니다. 이유는 세 객체 모두 차량이 실제로 반응하는 방향이 짧은 축과 더 잘 맞기 때문입니다.

  • 중심점: polygon 중심 [N_poly, 2]
  • 대표 방향: 짧은 축 방향 각도 [N_poly]
  • 로컬 좌표: 중심 기준 meter 값 그대로 사용
  • 크기값: 짧은 축 길이, 긴 축 길이만 사용

이렇게 두면 crosswalk는 차량 접근 방향, speed bump는 넘는 방향, driveway entrance는 진입 방향이 하나의 공통 축 의미로 정리됩니다.

2) boundary resampling

New / Fixed-Length Input

객체마다 점 개수가 달라도 encoder 입력은 고정 길이여야 하므로, 닫힌 경계를 둘레 기준으로 균일하게 다시 샘플링합니다. 권장값은 K = 8입니다. 이 정도면 직사각형 중심인 crosswalk / speed bump는 충분히 표현되고, driveway entrance의 비대칭도 어느 정도 유지됩니다.

  • 입력: raw polygon [N_poly, V_i, 2]
  • 출력: boundary points [N_poly, 8, 2]
  • 추가 계산: 연속 점 차이로 edge vector [N_poly, 8, 2]

이 단계 덕분에 원본처럼 4개 점만 뽑아 의미를 잃지 않고, 모든 polygon이 같은 크기의 입력으로 정렬됩니다.

3) boundary point encoder

MLP + 1-Layer Self-Attention

각 boundary point는 자기 위치만 보면 부족하므로, 다음 점으로 가는 경계 방향까지 함께 넣습니다. 그래서 한 점의 입력은 [x, y, dx, dy]입니다. 이 입력을 작은 MLP로 올린 다음, 경계 순서를 1층 self-attention으로 한 번만 섞어 줍니다.

  • 입력: [N_poly, 8, 4]
  • 점별 MLP: 각 점을 64차원으로 변환
  • 경계 순서 attention: [N_poly, 8, 64] 유지

여기서 self-attention을 1층만 넣는 이유는, 단순 pooling만 쓰면 경계 순서 정보가 약해지기 때문입니다. 하지만 층을 많이 쌓을 필요는 없습니다. 점 개수 자체가 작고, 파라미터도 아껴야 하기 때문입니다.

4) shape summary pooling

Mean Pool + Max Pool

점 수준 표현을 전체 객체 표현으로 바꾸는 단계입니다. mean pool은 전체 평균 모양을 담고, max pool은 가장 강한 경계 특징을 잡습니다. 두 벡터를 이어 붙이면 전체 shape summary가 됩니다.

  • 입력: [N_poly, 8, 64]
  • mean pool: [N_poly, 64]
  • max pool: [N_poly, 64]
  • 출력: [N_poly, 128]

5) size head + type head

Small MLP + Embedding

boundary 모양만으로는 crosswalk와 speed bump를 완전히 나누기 어렵기 때문에, 객체 고유 종류와 크기 요약을 별도 채널로 같이 넣습니다. 다만 입력 스케일 철학을 맞추기 위해, 큰 값이 될 수 있는 area는 직접 넣지 않고 짧은 축 길이, 긴 축 길이만 사용합니다.

  • size input: [N_poly, 2] = 짧은 축, 긴 축
  • size MLP 출력: [N_poly, 32]
  • type embedding 출력: [N_poly, 32]

이 정보는 shape summary와 합쳐져, 같은 직사각형처럼 보여도 crosswalk / speed bump / driveway를 다르게 해석하도록 도와줍니다.

6) fusion MLP

MLP

shape summary, size feature, type embedding을 이어 붙여 최종 초기 polygon token을 만듭니다. 이 단계까지의 출력은 아직 자기 모양만 아는 token입니다.

  • 입력: [N_poly, 128 + 32 + 32]
  • 출력: initial polygon token [N_poly, 128]

7) lane-conditioned refinement

1-Layer Cross-Attention

polygon은 자기 모양만 알면 부족합니다. 실제 주행에서는 어느 도로 흐름에 붙어 있는지가 중요합니다. 그래서 초기 polygon token이 주변 road token을 1층만 읽게 합니다. 이때 road token은 이미 기존 road branch가 잘 만든 표현을 그대로 씁니다.

  • query: initial polygon token [N_poly, 128]
  • key / value: road token [N_road, 128]
  • 관계 feature: [E, 3] = 거리, 상대 방향, heading 차이
  • 출력: refined polygon token [N_poly, 128]

이 1층 때문에 crosswalk는 어떤 주행 흐름 위에 놓였는지, speed bump는 어느 lane을 가로지르는지, driveway entrance는 어느 road edge / lane과 연결되는지가 token 안에 들어갑니다.

정리 polygon semantic encoder는 모양 자체는 boundary point encoder가, 객체 종류는 type embedding이, 크기 차이는 size head가, 도로와의 연결 의미는 lane-conditioned refinement가 담당합니다. 이 네 조합이면 지금 구조를 크게 어지럽히지 않으면서도 polygon 3종을 가장 분리해서 살릴 수 있습니다.

road / polygon / stop sign이 최종적으로 어떻게 합쳐지는가

여기서 중요한 점은 stop sign만 별도 token이 없다는 것입니다. stop sign은 lane type 2로 lane branch 안에 남고, polygon 3종만 따로 token이 생깁니다.

왜 stop sign을 안 건드리는가

stop sign은 지금 코드가 이미 lane semantics 안으로 넣고 있습니다. 이번 개정안에서 굳이 stop-sign 전용 token을 새로 만들면, 구조는 복잡해지고 파라미터도 늘어나는데, 핵심 약점인 polygon 3종과는 별개 분기가 또 하나 생깁니다. 이번 안은 그 유혹을 버리고, stop sign은 원본 유지로 고정합니다.

이번 안의 진짜 변경 포인트

결국 바뀌는 것은 딱 하나입니다. crosswalk / speed bump / driveway entrance가 road-like 선분이 아니라, 자기 의미를 가진 polygon token으로 살아난다는 점입니다. 나머지 문맥 trunk는 거의 그대로 유지됩니다.

Budget & Scaling

왜 이 안이 7M 수준과 현재 정규화 철학에 맞는가

이 프로젝트는 입력을 크게 표준화해서 넣는 스타일이 아니고, 상대값과 local 값 중심입니다. 그래서 polygon branch도 같은 철학으로 맞추는 것이 중요합니다.

정규화 원칙

polygon branch에서도 global normalization을 새로 만들지 않습니다. boundary point, edge vector, 짧은 축 길이, 긴 축 길이는 모두 local meter 값 그대로 넣습니다. heading은 기존 관계 feature와 맞게 angle 차이 또는 방향 벡터로 다룹니다.

왜 area를 직접 안 넣는가

면적은 숫자 스케일이 커질 수 있고, 현재 trunk와 감각이 다를 수 있습니다. 대신 boundary shape와 축 길이 두 개만 있어도 면 정보 대부분이 이미 들어갑니다.

파라미터 예산

새로 늘어나는 것은 작은 point MLP, 1층 boundary self-attention, 작은 size MLP, type embedding, 1층 polygon→road cross-attention 정도입니다. 그래서 증가분은 비교적 작고, 전체를 7M 근처에 유지하는 설계가 가능합니다.

왜 이게 가장 깔끔한가

road trunk, stop sign 처리, agent trunk, flow decoder는 그대로 두고, 약점이 큰 polygon 3종만 정교하게 살리기 때문입니다. 바꾸는 범위가 작고, 의미 분리는 커집니다.

최종 요약 road는 기존 road token bank를 계속 쓴다. stop sign은 기존 lane type 2 처리 그대로 둔다. crosswalk / speed bump / driveway entrance만 road bank를 우회하는 polygon semantic encoder로 만들고, 1층 polygon→road refinement 뒤에 기존 map→agent attention에 합친다. 이게 이번 조건에서 가장 단순하면서도 가장 합리적인 1안이다.
